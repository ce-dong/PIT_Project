# A 股 PIT 项目的简历导向优先级说明

## 1. 文档目的

本文档用于明确：这个项目的首要目标是服务于简历展示和实习面试，而不是建设一套长期自用、重运维的量化生产系统。

但“展示型项目”不等于“demo 风格项目”。

本项目在取舍上以简历展示为导向，在实现风格上仍然应坚持工业级的严谨性、规范性和可解释性，用以体现专业素养。

因此，本文档的核心原则不是“为了展示可以降低标准”，而是：

- 只做对展示价值高的部分
- 但凡决定要做，就按专业研究平台的标准去做

后续做功能取舍时，默认优先参考本文档。

## 2. 项目目标重心

这个项目最值得向面试官证明的，不是“我搭了一个每天自动运行的量化平台”，而是：

- 我理解中低频横截面研究的标准流程
- 我知道 `point-in-time` 的本质是避免前视偏差
- 我能把研究方法论落成结构化、可复现的工程底座
- 我不仅会算因子，还能说明数据口径、标签定义、评估框架和研究可信性
- 我能用规范、可审计、具备工程边界感的方式组织一个研究平台

换句话说，这个项目的展示价值主要来自：

- 研究方法论
- 数据口径严谨性
- PIT 底座设计
- 因子研究链路完整度
- 规范化工程能力
- 结果的可复现与可审计性

而不是来自：

- 复杂的调度系统
- 长期自动运维能力
- 生产级容错与监控
- 花哨但对研究主线帮助有限的工程包装

这里需要强调：

- 不追求重运维，不代表可以忽略规范
- 不追求大而全，不代表可以接受粗糙实现
- 不做生产系统，不代表可以放松对数据口径、测试、接口边界和文档的要求

## 2.1 工业级严谨风格的具体含义

本项目后续默认采用下列风格：

- 方法先行：重要口径先写清楚，再实现
- 数据可审计：关键时间字段、过滤条件、版本选择规则必须可追溯
- 接口稳定：Agent 1 和 Agent 2 通过稳定数据契约协作
- 可重复执行：同一输入下结果应尽量一致
- 失败可定位：日志、状态文件、分层模块要能支持排查问题
- 命名规范：字段、模块、表名、配置保持一致风格
- 测试覆盖关键风险点：优先覆盖前视偏差、时间规则和重复生成一致性

因此，本项目不追求“像工业系统那样臃肿”，但要追求“像专业团队产出那样规范”。

## 3. 最值得做的部分

### P0：必须完成，直接决定项目是否有简历价值

#### 3.1 Methodology Spec

必须有一份清晰的方法说明，至少能讲清楚：

- `rebalance_date` 如何定义
- `trade_execution_date` 如何定义
- `availability_date` / `tradable_date` 如何定义
- 财务字段如何避免前视偏差
- `universe` 的过滤规则
- forward return 的标签口径

原因：

如果这些说不清，项目就会从“研究平台”退化成“算几个因子的脚本集合”。

更进一步说：

如果这些规则没有形成文档化约束，即便代码能运行，也很难体现专业研究工作的严谨性。

#### 3.2 PIT 数据底座

Agent 1 的核心展示价值是搭出可信的 PIT 底座，至少要稳定产出：

- `calendar_table`
- `monthly_universe`
- `adjusted_price_panel`
- `monthly_snapshot_base`

原因：

这部分最能体现你不仅懂研究，也懂数据工程和研究平台抽象。

同时，这也是最容易体现“工业级严谨风格”的部分，因为：

- 需要明确时间规则
- 需要设计稳定接口
- 需要考虑重复生成和数据版本问题

#### 3.3 未来收益标签

必须有标准化标签层，至少包括：

- `fwd_ret_1m`
- `fwd_ret_3m`
- `fwd_ret_6m`

原因：

这是“平台是否真正能研究”的必要条件。

#### 3.4 标准因子评估链路

至少要有一套统一评估框架，能输出：

- Rank IC
- ICIR
- 分组收益
- top-bottom spread
- 覆盖率

原因：

这是从“数据项目”升级为“研究项目”的关键一步。

如果评估输出缺少统一口径、字段不规范或结果不可重复，展示效果会明显下降。

### P1：强烈建议完成，能显著提升项目讲述质量

#### 3.5 第一批经典因子

优先做最经典、最容易讲清楚的因子，而不是盲目扩大量。

建议优先：

- size
- book-to-market
- earnings-to-price
- 12_1 momentum
- 6_1 momentum
- turnover
- volatility
- ROE
- ROA
- asset growth

原因：

这些因子很适合面试交流，既经典，又能覆盖估值、动量、风险、质量、投资五条主线。

这里仍应坚持：

- 定义清楚
- 输入字段明确
- 滞后规则明确
- 评估口径统一

#### 3.6 自动化研究报告

至少要能输出一份结构统一的 markdown 报告，包含：

- 研究口径
- 因子定义
- 样本区间
- IC 和分组收益结果
- 结论摘要

原因：

这能大幅提升项目完成度，也方便你在简历和面试中展示“结果产物”。

报告不是简单截图堆砌，而应体现结构化输出能力。

#### 3.7 基础测试

至少覆盖：

- `rebalance_date` 生成正确
- `next_trade_date` 正确
- `availability_date <= tradable_date`
- `as-of join` 不前视
- 重复生成结果一致

原因：

面试时这类测试会显得你不是只会“跑通”，而是会验证研究底座可靠性。

这正是“展示型项目也保持工业级规范”的重要体现。

### P2：做了更好，但不要压过主线

#### 3.8 轻量 ingestion / update 能力

当前这部分已经有基本价值：

- 统一 `ingest` 入口
- 支持全量和增量
- 维护状态文件
- 有日志

对简历来说，这已经足够说明：

- 你不是一次性导数据
- 你考虑过平台的重复使用和数据维护
- 你理解工程化研究平台应具备最基本的数据更新闭环

这部分的定位应当是：

- 保持专业规范
- 但不追求重运维复杂度

因此，`ingest/update` 应该做到：

- 统一入口
- 明确全量与增量语义
- 保持状态可追踪
- 写入行为尽量幂等
- 出错时能定位问题

但不需要继续重投入做成：

- 调度平台
- 复杂监控
- 自动告警
- 分布式并发抓取框架

#### 3.9 更丰富的字段和因子

如果主线已经完成，再逐步扩展：

- `forecast`
- `express`
- 更多财务质量因子
- 行业中性化
- 市值中性化

但前提是不要影响 P0/P1 的完成质量。

## 4. 展示价值较低、应避免过度投入的部分

以下内容并非“没用”，而是对当前简历目标来说边际收益偏低。

重点不是“这些东西不专业”，而是“这些东西在当前阶段不是最能提升项目展示价值的专业投入”。

### 4.1 复杂自动调度

例如：

- `cron` 编排体系
- Airflow DAG
- Prefect 工作流
- 失败重试告警平台

这些更像生产系统建设，不是当前项目最应该优先展示的能力。

如果主线没有完成，先做这些会削弱项目叙事。

### 4.2 过重的工程基础设施

例如：

- 数据库迁移体系
- 多环境部署
- 服务化 API 层
- 权限系统

这些会把项目重心从研究平台拉偏。

规范化不等于基础设施堆砌。

### 4.3 过多低价值“因子 zoo”

如果只是为了把因子数量从 15 个堆到 80 个，但没有统一定义、评估和报告框架，反而会稀释项目质量。

### 4.4 为“看起来高级”而加入的复杂模型

例如：

- 深度学习选股模型
- 复杂集成学习系统
- 高维调参框架

这些容易让项目失焦，而且对你当前想展示的 PIT 横截面研究能力帮助有限。

## 5. 后续开发的默认决策原则

后面每做一个新功能，都先问下面 5 个问题：

1. 它是否能增强项目的“研究可信性”？
2. 它是否能增强项目的“平台完整性”？
3. 它是否能让简历描述或面试讲述更有说服力？
4. 它是否能体现规范、严谨、专业的实现风格？
5. 它是否会明显挤占主线时间？

如果：

- 前四项都弱
- 第五项很高

那默认不优先做。

## 6. 推荐的后续里程碑排序

从简历导向看，推荐优先级如下：

1. 完成 `calendar_table`
2. 完成 `adjusted_price_panel`
3. 完成 `monthly_universe`
4. 完成 `monthly_snapshot_base`
5. 完成 forward return 标签
6. 完成第一批经典因子
7. 完成统一评估框架
8. 完成 markdown 报告产出
9. 补基础测试和文档
10. 再考虑扩展字段、更多因子和轻量自动化

## 7. 面试导向的项目卖点表达

如果后续项目按本文档推进，最终比较适合提炼成下面几类卖点：

- 搭建 A 股 `point-in-time` 月频横截面研究平台，解决公告可得时间和无前视偏差问题
- 基于 Tushare 构建原始数据落地、标准化、PIT 对齐和月频 snapshot 底座
- 设计标准化的因子、标签和评估框架，支持 IC、分组收益和稳健性分析
- 通过统一数据契约和中间表，把上游数据工程与下游研究评估解耦
- 在简历导向项目中仍保持规范化文档、测试、日志与可复现流程，体现专业研究工程素养

## 8. 对后续实现的约束

从本文档开始，后续优先遵循以下原则：

- 优先研究口径和 PIT 可信性，而不是花哨工程
- 优先可复现和可解释，而不是过度自动化
- 优先“少而完整”的因子与评估链路，而不是堆很多功能点
- 优先能形成简历亮点和面试故事的部分
- 已经决定做的模块，默认按专业、规范、严谨的标准完成
- 不因为“只是展示项目”而放松命名、测试、文档和接口边界要求
